02 — ISOLATION

두 guard는 직교한다 (충돌 불가)

시스템엔 별개의 두 결정적 가드가 있다. 하나는 파일시스템 접근을, 다른 하나는 논증 그래프 무결성을 본다. 검증에서 확인한 핵심: 둘의 검사 축이 서로 직교해서 절대 부딪히지 않는다.

04 — BOTTOM-UP CHAIN

승격 사슬: 흔적 → 스킬 → 정본

반복되는 패턴은 자동으로 다음 단계 승격이 제안된다. 하지만 제안은 강제가 아니다 — judgment를 거쳐 owner가 닫는다. 파란 단계는 휴리스틱(제안), 황색 단계는 결정론(게이트).

1

워커 흔적 휴리스틱

워커가 자기 .context/task-log/에 작업을 자동 기록. 여러 워커가 같은 시그니처를 반복하면 탐지기가 후보로 표면화.

2

승격 후보 (SessionStart) 휴리스틱

detect_team_promotions가 read-only로 핸드오프 구조를 읽어 team_skill·project_skill·canon_promote 후보를 띄움. 레코드를 만들지 않는다.

3

owner 결정 (resolve) 게이트

팀장/orchestrator만 resolve --decision promote. 결정 메타데이터만 기록 — canon 레코드 자동저작은 어서션으로 차단(F-D5).

4

정본 저작 게이트

owner가 별도 Write 행위로 .project/evidence/ 등에 레코드 저작. 여기서만 canon_integrity가 링크를 검사하고 exit 2로 게이트.

05 — THE F-D5 INSIGHT

결정론과 휴리스틱은 왜 안 부딪히나

"자동승격"(반복 수치가 전부 canon이 됨)과 "강제승격"(깨진 링크면 차단)은 동시에 성립할 수 없다 — 같은 행위에 겹칠 때만. 토글을 눌러 두 양식이 시간축에서 어떻게 분리되는지 보라.

▷ 휴리스틱 — SessionStart
① 여러 워커가 같은 수치 반복 인용
② 후보로 표면화 (read-only)
③ owner가 judgment로 resolve
→ exit 0, 레코드 안 만듦
▷ 결정론 — PreToolUse
④ owner가 별도 Write로 레코드 저작
⑤ canon_integrity가 링크 검사
⑥ 깨졌으면 exit 2 차단
→ 성하면 통과
