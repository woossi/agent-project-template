# 스킬: manuscript-review-gate

## 사용 시점

manuscript-writer(또는 section-writer)가 원고 섹션·파트를 재작성·수정하고 검수요청을 보냈을 때, 그 산출물을 독립 품질 게이트로 판정해야 할 때 사용한다. 구체적으로:

- 집필자가 적용했다고 보고한 변경분(P0 수정·섹션 재작성)을 재판정(R1, R2, …)할 때.
- 이전 라운드 결함(F-id)의 해소 여부를 close 판정할 때.
- 새 결함을 발견해 등급(MAJOR/MINOR)과 재작업 지시를 산출할 때.

집필은 하지 않는다. 검수의견·체크리스트·재작업지시만 산출한다. 일회성이 아닌 반복 게이트 절차이므로 스킬로 둔다.

## 목적

집필자의 자기검수를 대체하는 독립 검수 게이트를 일관된 형식으로 운영한다. inbox 보고를 신뢰의 근거로 삼지 않고 산출물을 직접 대조하며, 지적된 항목을 넘어 그 항목이 속한 단위 전체의 무결성까지 검증해 결함을 한 라운드라도 일찍 발견한다.

## 계약

- 읽는 입력: 집필자 inbox 검수요청(변경 보고), 검수 대상 원고(작업 경계 내 `*.tex`·표·그림·refs), 직전 라운드 체크리스트(F-id 미해소 목록), 도착 시 stats-validator 수치검증.
- 만드는 출력: `.context/review-checklist.md`의 재판정 R<n> 섹션(판정표·F-id 결함·재작업 지시), 미리알림 annotate, 집필자·orchestrator inbox reply.
- 쓰면 안 되는 위치: 원고 파일(집필자 소관 — 직접 수정 금지, 수정안 예시만 제시). 형제 에이전트 폴더(`agents/<다른이름>/`). 공유 store는 CLI(Bash)로만 접근.

## 입력

- 집필자 inbox 메시지의 변경 보고와 원본 msgid(reply-to용).
- 검수 대상 원고 경로(예: `/Users/ujunbin/research/UMC/parts/*.tex`, `tables_main.tex`, `umc_paper.tex`).
- 직전 라운드 `.context/review-checklist.md`(미해소 F-id).
- 게이트 기준: A(저널 투고요건) / B(정합성·완결성) / C(방법론 기여 Q1 정렬) / D(통계 수치) / **E(관통논리: 서술의 지위 — `.team/memory.md` `manuscript-status-of-claims` 기준)**. 본 검수자 체크리스트 v1의 항목 ID 체계를 따른다.

## 절차

1. **inbox 보고를 검증 대상으로만 취급한다.** "적용·보존 완료" 보고는 판정 근거가 아니라 검증해야 할 가설이다. 보고된 라인·파일을 직접 Read/grep로 대조한다.
2. **직전 F-id close 판정 먼저.** 미해소 결함이 실제로 고쳐졌는지 산출물에서 확인하고, 집필자가 든 근거(예: "표본 희박")는 데이터(표 셀값)로 역검증한다. 해소면 CLOSE, 미해소면 잔존 사유 명시.
3. **단위 전체 수치 무결성 검산(핵심).** 지적·변경된 항목만 보지 말고, 그 항목이 속한 절·표에 등장하는 **모든 합산 가능 수치 집합을 한 번에 검산**한다. `grep`으로 수치를 추출해 `python3`로 부분합=총계, 표↔본문↔초록↔그림 캡션 일치를 대조한다. 부분합이 총계와 어긋나면 그 부분합이 무엇의 하위집합인지 본문이 정의하는지 확인하고, 정의가 없으면 결함으로 기록한다(집필자가 못 본 결함 발견이 검수자의 1차 가치).
4. **게이트 축별 판정.** A/B/C(필요 시 D) 축으로 PASS / PARTIAL / FAIL / N/A를 매긴다. A1(영어)·A3(초록 길이) 등 하드 게이트 미통과는 다른 항목과 무관하게 자동 FAIL. **문체·수사 품질**은 `references/writing-quality-rubric.md`의 다차원 지표(구조·수사 / 구체성·정량성 / AI·상투 문체 / 불확실성·정직성)로 측정한다. 특히 방어적 과잉 hedge(D2)와 미검증분 hedge 의무(D1)를 짝으로 보아 "hedge 유무가 아니라 대상 정합성"을 판정한다.
4b. **관통논리(서술의 지위) 판정 — E축.** 각 섹션의 한정·단서·해명을 세 종류로 분류하고 종류별 올바른 동작이 적용됐는지 판정한다(근거: `.team/memory.md` `manuscript-status-of-claims`). 이 한정은 (1)/(2)/(3) 중 무엇인가?
   - **(1) 설계요구**(min-max 상대척도·자치구 단위·정보차단 멀티에이전트·예측오차 우선탐색) → **긍정 선언**인가? "차선/대용 아님"식 부정 경유면 위반(=(3) 위장).
   - **(2) 데이터 한계**(진짜 한계 6종) → §5에 사실로 **1회** 적시됐나? 삭제·단정화면 위반(**과소**).
   - **(3) 선제사과 항변**(존재론 동일성 부인·"발견법으로만"·"등가 아님" 반복, "탐색적 윤곽으로만" 반복) → **삭제**됐나? 같은 동작 2회+ 잔존이면 위반.
   - **(2) 진짜 한계 6종 §5 1회 보존 확인**(과소=위반, Clarke steel-person): ①선택편향 비식별성(당근 이용자=디지털 접근가능층) ②단일 통신사(SKT) 대리지표 ③낮은 ICC(0.49%) 2수준 검정력 제약 ④동일출처 오염회피 위한 차원제외 대가 ⑤플랫폼 비대표성(하한값·선택적 증거) ⑥min-max 절대충분성 비판단(상대위치이지 절대결핍/충분 아님). 6종 중 §5에서 사라진 게 있으면 한계 은폐로 FAIL.
   - **반복항변 카운트**: 같은 항변 2회+ 잔존이면 `R<n>-F` 기록(intro·ch2·ch4 가교의 등가부정, results의 탐색적 반복헤지가 단골). friston2010·parr2019 등 가교 인용은 항변 삭제 시 "중립 방법 출처"로 1회만 허용.
   - **53계열 특례**: (3) 메타해명("경계 불확실/탐색적 윤곽으로만")은 걷되, "53이 불확실하다는 사실"은 (2)이므로 [불확실] 표기 유지가 맞다(임의 보정·재집계는 위반). 본문 미제시도 [불확실] 정합 — 표 tab:stage_e 126계열 불변이 조건.
   - **판정**: E축 PASS/PARTIAL/FAIL, 근거는 파일·라인. D축과 짝(D=hedge 대상 정합, E=hedge가 (1)/(2)/(3) 중 무엇인지). D2(과잉hedge)와 (3)삭제가 겹치고 D1(미검증분 hedge 의무)과 (2)보존이 겹친다 — 모순 없는 보강.
4c. **출력 무결성 실측 — 빌드 PASS ≠ 출력 무결성.** LaTeX 빌드가 PASS(undefined citation 0, `\ref`↔`\label` 정합)여도 **출력물 무결성을 단정하지 않는다.** scratchpad에서 빌드한 PDF를 `pdftotext`로 평문 추출해 비-LaTeX 마크업·오염 토큰(예: `</content>`, 잔존 XML/HTML 태그, 에디터 프롬프트 부산물)이 출력에 살아남았는지 grep으로 실측한다. 발견 시 MINOR 이상으로 `R<n>-F` 기록(R9-F1 교훈: 빌드 PASS·검수자 직접 발견). 출력 텍스트 길이·문자수 점검으로 분량 게이트(A2/A3) 보조 검증도 겸한다.
5. **결함에 F-id·등급 부여.** `R<n>-F<k>` 식별자, MAJOR(재작업 반려)/MINOR(정리 권고), 파일·라인·요구사항, 그리고 수정안 예시(적용은 집필자)를 단다. 수치 결함은 임의 보정 금지 — 실측 확정(stats-validator 등) 연계를 지정한다.
6. **체크리스트에 재판정 R<n> 섹션 추가.** 판정표 + F-id 상세 + 종합(PASS 확정·반려 목록·미착수·미도착 의존). 직접 집필하지 않는다는 게이트 운영 규칙을 지킨다.
7. **라우팅.** 미리알림 해당 작업 노트에 `annotate`로 판정 요약, 집필자·orchestrator(수치 결함이면 stats-validator 포함)에 `--reply-to`로 reply. 처리한 메시지는 `ack`. 동일 검수요청이 중복 라우팅되면 직전 회신 msgid를 들어 처리완료만 통지.

## 출력 형식

- `.context/review-checklist.md`의 `## [재판정 R<n>] <대상> (<날짜>)` 섹션:
  - 판정표 `| 항목/축 | 판정 | 근거(직접 검증) |`
  - F-id 상세 `### R<n>-F<k> (<MAJOR|MINOR>, <요지>)` — 결함·근거·재작업 지시·수정안 예시
  - `### 재판정 R<n> 종합` — F-id close 여부, PASS 확정, 반려 목록, 미착수·미도착 의존
- 미리알림 annotate 1줄 요약(판정·F-id·근거 위치).
- inbox reply(집필자·orchestrator·필요 시 stats-validator).

## 내부 자원

- `references/` — 검수 판정 기준 자원 폴더
- `references/writing-quality-rubric.md` — 학술 글 품질 검수 다차원 rubric(구조·수사 / 구체성·정량성 / AI·상투 문체 / 불확실성·정직성 4축, 0–2 채점). econ-writing-skill(Cochrane·McCloskey 등 50+ 가이드 종합)을 검수자 관점으로 내재화·팀 정책 조정.
- 수치·정합 검산은 세션의 Bash(grep/python3)와 team-inbox·reminders-team-bridge 스킬 CLI를 호출한다. 보조 스크립트를 추가하면 이 절에 적는다.

## 품질 점검

- 모든 PASS/CLOSE 판정에 **산출물 직접 검증 근거**(파일·라인 또는 검산식)가 달려 있어야 한다. inbox 보고 인용만으로 PASS를 내지 않는다.
- 수치 결함 보고에는 실제 검산식(부분합 vs 총계)이 포함되어야 한다.
- 원고 파일을 직접 수정하지 않았어야 한다(수정안 예시는 허용).
- 모든 F-id에 등급·파일·라인·재작업 지시가 있어야 한다.
- 처리한 inbox 메시지가 ack되고, 미리알림 annotate와 reply가 발신되었어야 한다.

## 자주 발생하는 실패 사례

- **지적 항목만 보고 절 전체 수치를 안 봄** → 그 절의 합산 가능 수치를 모두 한 번에 검산한다(R1에서 line77만 보다 R2에서야 53 vs 126 불일치를 발견한 사례의 교훈).
- **inbox "보존 완료" 보고를 PASS 근거로 인용** → 보고는 가설일 뿐. 산출물에서 직접 검산해야 PASS.
- **수치 불일치를 임의 보정값으로 메움** → 금지. 실측 확정(stats-validator) 연계 후 집필자 적용.
- **집필자 핸드오프 로그를 읽으려다 형제폴더 guard 차단** → 정상. 보고 핵심은 inbox 본문에 있고, 검증은 작업 경계 내 원고로 한다.
- **검수자가 원고를 직접 고침** → 게이트 규칙 위반. 수정안 예시만 제시하고 적용은 집필자가 한다.
- **(1) 설계 귀결을 (3) 항변으로 위장한 채 E축 PASS** → "X는 차선이 아니다"식 부정 경유는 (3)이다. 긍정 선언("X는 목적이 요구한 설계 귀결")으로 바뀌어야 PASS. 부정 경유로 남아 있으면 (3) 위장으로 위반 기록.
- **(2) 진짜 한계를 단정화·삭제하고 E축 PASS** → 6종 중 §5에서 사라진 한계는 은폐다(긍정 선언 위장이 한계를 가리면 더 나쁜 상태). 과소도 위반으로 FAIL.
- **빌드 PASS만 보고 출력 무결성 PASS** → 빌드 통과(undefined 0)는 출력 무결성을 보장하지 않는다. pdftotext로 PDF를 실측하지 않으면 `</content>` 같은 마크업 잔존을 놓친다(R9-F1: 빌드 PASS였으나 출력 오염, 검수자 직접 발견). 4c 단계로 실측한다.

## 계약 연계

- 스킬은 반복되는 작업 묶음을 하나의 포괄 이름으로 승격한 재사용 절차다.
- 작업은 이 스킬을 `사용할 스킬` 입력으로 참조한다. 절차를 복사하지 않는다.
- 서브에이전트는 작업 입력을 받아 필요한 스킬 능력을 사용한다.
- 현재 진행상황과 handoff는 스킬에 저장하지 않는다.
