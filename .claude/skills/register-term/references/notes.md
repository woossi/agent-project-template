# 용어 등록 참고 메모

`register-term` 스킬의 보조 자료. 무엇을 등록할지, 각 필드를 어떻게 쓸지에 대한 기준.

## 등록할 것 / 등록하지 않을 것

**등록한다**
- 프로젝트에서 반복 등장하고, 사람마다 다르게 이해할 여지가 있는 약어·전문어.
- 한국어/원어 표기가 섞여 혼동되는 번역어(예: "임베딩" vs "embedding").
- 합의가 필요한 도메인 용어(정의를 한 곳에 고정해 두면 이득이 큰 것).

**등록하지 않는다**
- 한 번 쓰고 말 일회성 단어, 그 자리에서 설명하면 끝나는 것.
- 일반 상식 수준의 보통명사.
- 진행 상황·결정 같은 기록(이건 `memory.md`/`tasks.md`의 몫).

## 필드 작성 지침

| 필드 | 의미 | 좋은 예 | 피할 것 |
| --- | --- | --- | --- |
| `term` | 원어/표준 표기 | `RAG`, `LISA`, `choropleth` | 임의 약자, 표기 흔들림 |
| `ko` | 한국어 표현 | `검색 증강 생성`, `단계구분도` | 원어 그대로 음차만 |
| `definition` | 한 문장 정의 | "외부 지식을 검색해 LLM 응답에 결합하는 기법" | 순환 정의("RAG는 RAG다"), 여러 문장 |
| `use_when` | 쓰는 맥락/조건 | "출처가 필요한 답변을 만들 때" | 비워 두기, "항상" |

- 4개 필드는 모두 필수다. 하나라도 모호하면 등록하지 말고 사용자에게 물어 확정한다.
- `term`은 사전 내에서 고유해야 한다(대소문자 무시). 같은 개념의 변형 표기는 하나로 통일한다.

## 자주 하는 실수

- 정의를 추측해서 채움 → 금지. 모호하면 묻는다.
- `term` 표기를 그때그때 다르게 적어 중복 항목이 생김 → 표준 표기 하나로 고정.
- 4개 필드를 다 채우지 않고 추가 시도 → 스크립트가 종료코드 1로 거부한다.
