## Aesthetic Guidelines
# NeurIPS 2025 Method Diagram Aesthetics Guide

## 1. The "NeurIPS Look"

The prevailing aesthetic for 2025 is "Soft Tech & Scientific Pastels."
Gone are the days of harsh primary colors and sharp black boxes. The modern
NeurIPS diagram feels approachable yet precise. It utilizes high-value (light)
backgrounds to organize complexity, reserving saturation for the most critical
active elements. The vibe balances clean modularity (clear separation of parts)
with narrative flow (clear left-to-right progression).

## 2. Detailed Style Options

### A. Color Palettes

Design Philosophy: Use color to group logic, not just to decorate.
Avoid fully saturated backgrounds.

Background Fills (The "Zone" Strategy):
- Most papers use very light, desaturated pastels (Opacity ~10-15%).
- Aesthetically pleasing options: Cream/Beige (warm academic feel),
  Pale Blue/Ice (clean technical feel), Mint/Sage (soft organic feel),
  Pale Lavender (distinctive modern feel).
- Alternative (~20%): White backgrounds with colored dashed borders.

Functional Element Colors:
- For "Active" Modules: Medium saturation. Common pairings: Blue/Orange,
  Green/Purple, or Teal/Pink.
- Trainable Elements: Often Warm tones (Red, Orange, Deep Pink).
- Frozen/Static Elements: Often Cool tones (Grey, Ice Blue, Cyan).
- For Highlights/Results: High saturation reserved for "Error/Loss,"
  "Ground Truth," or final output.

### B. Shapes & Containers

Design Philosophy: "Softened Geometry." Sharp corners for data; rounded
corners for processes.

- Process Nodes: Rounded Rectangles (dominant ~80%).
- Tensors & Data: 3D Stacks/Cuboids for volume, Flat Squares/Grids
  for matrices/tokens.
- Cylinders: Reserved for Databases, Buffers, or Memory.
- Grouping: Solid light-colored containers with "Macro-Micro" pattern.
- Borders: Solid for physical components, Dashed for logical stages.

### C. Lines & Arrows

- Orthogonal/Elbow: For Network Architectures.
- Curved/Bezier: For System Logic, Feedback Loops.
- Solid Black/Grey: Standard data flow.
- Dashed Lines: Auxiliary flow (gradients, skip connections).
- Integrated Math: Operators placed directly on lines.

### D. Typography & Icons

- Labels: Sans-Serif (Arial, Roboto, Helvetica). Bold for headers.
- Variables: Serif, Italicized (LaTeX style).
- Icons: Trainable (Fire, Lightning), Frozen (Snowflake, Padlock),
  Operations (Gear, Magnifying Glass), Content (Document, Chat Bubble).

### E. Layout & Composition

- Flow: Left-to-right for sequential, top-to-bottom for hierarchical.
- Alignment: Snap to implicit grid.
- Spacing: Consistent gaps, closer within groups.
- Balance: Distribute visual weight evenly.
- Whitespace: Intentional separation of phases/concepts.

## 3. Common Pitfalls

- The "PowerPoint Default" Look with heavy black outlines.
- Font Mixing (Times New Roman for labels).
- Inconsistent Dimension mixing (2D and 3D without reason).
- Primary saturated backgrounds for grouping.
- Ambiguous arrows (same style for data flow and gradient flow).

## 4. Domain-Specific Styles

- AGENT/LLM Papers: Illustrative, narrative, cartoony. Chat bubbles,
  robots, document icons.
- COMPUTER VISION Papers: Spatial, dense, geometric. Frustums, rays,
  point clouds, RGB color coding.
- THEORETICAL Papers: Minimalist, abstract. Graph nodes, manifolds,
  grayscale with one highlight color.
- GENERATIVE/LEARNING Papers: Dynamic, process-oriented. Noise/denoising
  metaphors, gradual color transitions.

*   **System Output**: Supported final coding

### Grouping: Layer 1: Classification
*   **Unit of Analysis**: post

*   **Sequential Operations**:

        *   **Input**: Candidate posts

    3.  **Component**: Dimension tagging
        *   **Input**: Posts with relevance codes

### Grouping: Layer 2: EB Aggregation
*   **Unit of Analysis**: district x dimension
*   **Description**: Aggregates classified posts, computes rates, and applies empirical Bayes shrinkage.
*   **Input**: Classified post base (from Layer 1)
*   **Sequential Operations**:
    1.  **Component**: Dong linkage
        *   **Input**: Classified post base
        *   **Output**: Posts linked to dong (administrative units)
    2.  **Component**: Exposure denominator calculation
        *   **Input**: Posts linked to dong
        *   **Output**: Exposure denominator (for rate calculation)
    3.  **Component**: Observed post rate calculation
        *   **Input**: Posts linked to dong, Exposure denominator
        *   **Output**: Observed post rate
    4.  **Component**: EB shrinkage (Empirical Bayes shrinkage rate model)
        *   **Input**: Observed post rate, Administrative prior (implied)
        *   **Mathematical Operation**: Combines administrative prior with platform observations using an empirical Bayes shrinkage rate model.
        *   **Output**: Posterior rate
    5.  **Component**: High-z_shift review cells flagging
        *   **Input**: Posterior rate
        *   **Output**: High-z_shift review cells
*   **Layer Output**: High-z_shift review cells (data type: cells flagged for review)

### Grouping: Layer 3: Evidence-grounded Judgment
*   **Unit of Analysis**: post-level case
*   **Description**: Runs five execution stages (A-E) across an information boundary to produce a final coding.
*   **Input**: High-z_shift review cells (from Layer 2)
*   **Key Relationship**: Information Boundary (drawn as two dashed boxes) separates I_R and I_J. This boundary blocks confirmation bias.
*   **Key Relationship**: High-divergence (high z_shift) cells trigger post-level hypotheses but do not determine the final judgment.

#### Sub-Grouping: I_R: Stage A/B Isolated Inputs (Restricted Information Set)
*   **Description**: Hypothesis generation stage with restricted information access.
*   **Input**: Stage A post record (triggered by High-z_shift review cells)
*   **Parallel Operations (Stage B)**:
    1.  **Component**: Abductive hypothesis agent
        *   **Input**: Stage A post record (post text/metadata/dimension codes)
        *   **Restriction**: Does NOT see EB divergence or district profiles.
        *   **Output**: Abductive hypothesis
    2.  **Component**: Forward hypothesis agent
        *   **Input**: Stage A post record (post text/metadata/dimension codes)
        *   **Restriction**: Does NOT see EB divergence or district profiles.
        *   **Output**: Forward hypothesis
    3.  **Component**: Sequential hypothesis agent
        *   **Input**: Stage A post record (post text/metadata/dimension codes)
        *   **Restriction**: Does NOT see EB divergence or district profiles.
        *   **Output**: Sequential hypothesis
*   **Output**: Set of generated hypotheses (from Abductive, Forward, Sequential agents)

#### Sub-Grouping: I_J: Stage C/D/E Evidence Set (Expanded Judgment Set)
*   **Description**: Judgment stage with expanded evidence access.
*   **Input**: Hypotheses (from I_R, crossing the information boundary – "hypothesis hand-off")
*   **Key Relationship**: Only judgment (I_J, Stage C/D/E) sees expanded evidence.
*   **Sequential Operations**:
    1.  **Component**: Stage C data specification
        *   **Input**: Hypotheses
        *   **Output**: Data specification based on hypotheses
    2.  **Component**: Stage D rule-based evidence
        *   **Input**: Data specification
        *   **Output**: Rule-based evidence
    3.  **Component**: Stage E judgment
        *   **Input**: Rule-based evidence
        *   **Output**: Supported final coding
*   **Layer Output**: Supported final coding (data type: final classified post coding)

This multi-layered flowchart illustrates the three-layer UMC pipeline, depicting a top-to-bottom flow. Layer 1 (Classification) processes platform records to a "Classified post base," which feeds into Layer 2 (EB aggregation) to compute rates, apply empirical Bayes shrinkage, and identify "High-z_shift review cells."
